Findings:

- `src/Cardano/Testnet/Baker/Synthesis.hs:227` — `chainDbFileEntry`
  reads each chain-db file fully into memory via `LBS.readFile`
  and forces it to strict via `LBS.toStrict bytes` before digesting.
  That is fine for `local-fast` (720 slots, kilobytes per chunk),
  but the realistic `normal` measurement at `slotCount = 300000`
  produces immutable chunks in the hundreds of megabytes; one or
  more such files in flight will spike resident memory. Either:
  - stream the digest with a chunked hash builder (e.g. via
    `cryptonite` `hashUpdates`), or
  - file an explicit follow-up task (e.g. T036) to revisit before
    the realistic measurement pipeline is exercised on CI.
  Document whichever path you take in the body so the next reader
  knows it is a known limit, not an oversight.
- `src/Cardano/Testnet/Baker/Bake.hs:329` — `host` is read from
  `lookupEnv "HOSTNAME"`. On most Linux distributions `HOSTNAME` is
  a shell-only variable (not exported into the process env), so
  the field will read `"unknown"` in the common case despite a
  well-known host being available. Use
  `Network.HostName.getHostName` from the `network` package (or
  `Network.BSD.getHostName`) — both return `IO String` and read
  the kernel's hostname directly. The field stays in the
  host-dependent `observation` block, so changing the source has
  no determinism impact.
