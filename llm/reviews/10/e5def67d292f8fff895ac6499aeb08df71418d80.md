Findings:

- **Recorded measurement evidence is implausibly small.** The
  ready file and `specs/002-chaindb-synthesis/quickstart.md`
  pin `chainDb.bytes=5`, `chainDb.fileCount=4`,
  `chainDb.packagedBytes=488` for a 300k-slot synthesis of
  `normal.json`. Five bytes of ChainDB across four files is not
  what a real 300k-slot upstream `db-synthesizer` run produces;
  it looks like the synthesizer wrote configuration/manifest
  files and produced no actual blocks (e.g. because of an
  `epochLength`/`securityParam`/`activeSlotsCoeff` combination
  that yields empty epochs at this scale). Either:
  - re-run with parameters that exercise the realistic
    measurement path the task describes (T035 is supposed to
    *measure storage and timing for the 300k-slot case*; 488
    bytes packaged is not a meaningful storage signal), or
  - if these numbers are the correct steady-state for the
    current `normal.json` shape, document explicitly in
    `quickstart.md` that the recipe runs a degenerate
    measurement and that `normal.json` needs an upstream-
    constraint adjustment (analogous to the `local-fast`
    `securityParam=2` work) before T035 is considered done.
  Approving these numbers as-is would mean shipping a
  measurement workflow that does not actually measure.
- `nix/checks.nix:73` — the report-shape predicate accepts
  `chainDb.bytes >= 0`, so the same degenerate output is
  considered valid. Once the measurement parameters are
  corrected, tighten the local-fast lower bound to a
  non-trivial floor (e.g. `> 0` and `fileCount >= 3` for the
  three immutable/ledger/volatile subdirectories) so a future
  regression that drops back to a near-empty ChainDB fails the
  gate.
