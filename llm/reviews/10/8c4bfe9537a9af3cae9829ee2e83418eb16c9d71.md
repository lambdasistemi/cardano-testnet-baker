- commit message — title: `feat(scenario): decode synthesis request from scenario JSON`
- commit message — body:
  > Adds the `SynthesisRequest` record (`enabled`, optional `slotCount`,
  > optional `profile`) and threads it through `Scenario` as
  > `scenarioSynthesis :: Maybe SynthesisRequest`. The Haskell type is
  > deliberately a flat record with two `Maybe` fields rather than the
  > schema's `Disabled | Enabled SlotCount _` shape; the
  > `slotCount required when enabled` invariant is enforced upstream by
  > schema validation, not in the parser. Document-level validation
  > and a tightened type land in a follow-up task.

Findings:

- `src/Cardano/Testnet/Baker/Scenario.hs:88` — `synthesisSlotCount ::
  Maybe Int` permits `{enabled = True, slotCount = Nothing}` to parse,
  which the schema rejects. The body addition above documents this as
  "enforced upstream by schema validation"; if that is *not* the
  intent, replace the record with a sum type
  (`data SynthesisRequest = Disabled | Enabled Int (Maybe Text)`) so
  the invariant is unrepresentable. Either way the call sites that
  consume `SynthesisRequest` need to know which guarantee they have —
  flag this in the body or in a tasks.md follow-up.
- `test/Cardano/Testnet/Baker/ScenarioSpec.hs:46` — only one positive
  test for synthesis decoding. Add at least:
  - a fixture where `synthesis` is omitted asserting
    `scenarioSynthesis == Nothing`,
  - a fixture with an unknown key under `synthesis` asserting decode
    failure (exercises `rejectUnknownKeys`),
  - a fixture where `synthesis.enabled` is missing asserting decode
    failure.
  These are the minimum-viable negative cases for a new decoder; the
  current single happy-path test does not protect against regressions
  in the rejection paths.
