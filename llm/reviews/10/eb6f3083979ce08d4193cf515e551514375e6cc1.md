Findings:

- `test/Cardano/Testnet/Baker/BakeSpec.hs` — the new `staking.pools`
  and `staking.stake` populated paths are not directly asserted. Add
  a test that decodes the rendered Shelley genesis for a pool-bearing
  scenario and asserts:
  - `.staking.pools` has one entry per declared pool, keyed by the
    pool's cold key hash;
  - each entry's `.publicKey` matches `poolColdKeyHashHex` and
    `.vrf` matches `poolVrfKeyHashHex` for the same pool;
  - `.staking.stake` maps each pool's stake key hash to its cold key
    hash;
  - `.initialFunds` contains the pool stake address with
    `poolStake pool` lovelace.
  Without this coverage, a later refactor that drops the registration
  silently regresses synthesis to the same degenerate state that
  motivated this rollback.
- `src/Cardano/Testnet/Baker/Keys.hs:191` — `poolStakeAddressHex`
  uses `PoolStakePaymentKey` with `poolStakeKeyLabel pool` as the
  derivation label. The label is reused across two roles
  (`PoolStakeKey` and `PoolStakePaymentKey`); the role byte
  separates them in `deriveSigningKey`'s seed expansion, so this is
  safe, but it is non-obvious. Add a one-line comment on
  `poolStakeAddressHex` recording why the same label is intentionally
  reused — i.e. that the role tag in the derivation prevents
  collision and keeps the address deterministic per pool.
