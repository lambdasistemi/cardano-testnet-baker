- commit message — body:
  > Closes the type/schema gap from the prior patch by validating
  > `SynthesisRequest` semantically: `slotCount` is required when
  > `enabled = true`, `slotCount` must be positive, and `profile` (when
  > given) must be non-empty. Failures aggregate alongside the existing
  > pool/faucet/duplicate-label checks rather than short-circuiting, so
  > callers receive the full set of issues per scenario.

  Run `stg edit -m "feat(validation): enforce synthesis invariants in scenario validation" -m "<body>"`.

Findings:

- `test/Cardano/Testnet/Baker/ScenarioSpec.hs` — no positive validation
  test for `enabled = true` with valid `slotCount` and `profile`. The
  existing tests cover `Nothing`, `enabled=false`, and three failure
  paths, but a regression that breaks the happy path on enabled
  synthesis would not be caught. Add a single test reusing or mirroring
  `scenarioWithSynthesis` and asserting
  `validateScenario scenario == Right (ValidatedScenario scenario)`.

Implementation otherwise sound: failures aggregate via list
concatenation, `SynthesisSlotCountNotPositive` carries the offending
value for downstream error messages, and the validator runs
unconditionally on `Just slotCount`/`Just profile` so the same checks
apply to disabled synthesis (mirrors the schema's unconditional
`minimum`/`minLength` constraints).
