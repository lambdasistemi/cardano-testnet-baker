### What

`nix/seed-publish.nix` (new):

- Exports `publishSeedImages` as a
  `pkgs.writeShellApplication`. The runtime closure
  declares `coreutils`, `gnutar`, `gnugrep`, `gnused`,
  `jq`, `skopeo` so the app is hermetic when invoked via
  `nix run`.
- Iterates `scenarioFiles` (the same set
  `seedImagePackages` and `seed-image-acceptance`
  enumerate) at flake-eval time and references the
  per-scenario archive built by `seedImage.mkSeedImage`.
  Adding a new `examples/scenarios/<name>.json` extends
  the publish coverage automatically.

The script, per scenario:

1. `extract_scenario_digest` — runs
   `skopeo copy "docker-archive:$archive" "dir:..."`,
   parses the resulting `manifest.json` for the single
   layer, untars it, and returns
   `metadata.json.inputDigest`. Per
   [contracts/artifact-identifier-scheme.md](../specs/003-seed-distribution/contracts/artifact-identifier-scheme.md)
   the `inputDigest` field is the canonical source of the
   consumer-facing `<scenarioDigest>` tag fragment;
   reading it from the *built* image (rather than the
   unpackaged bake) means CI cannot accidentally publish
   under a tag that doesn't match what the consumer
   downloads.
2. Derives `primary = <scenario>-<scenarioDigest>` and
   `secondary = <scenario>-sha-<commit-sha-short>` per
   the artifact-identifier-scheme contract.
3. `validate_tag` runs both a literal block list
   (`latest`, `main`, `master`, `next`, `dev`,
   `prod`) and a positive grammar check (every tag must
   end in either `-<64-hex>` for the primary form or
   `-sha-<7-hex>` for the secondary form). Refusal is
   loud (`stderr`, exit 1) and happens *before* any
   `skopeo copy` invocation.
4. `skopeo copy --src-tls-verify --dest-tls-verify
   docker-archive:<archive>
   docker://ghcr.io/lambdasistemi/cardano-testnet-seed:<tag>` —
   primary first, then secondary. Per contract
   [`publish-pipeline.md §"Failure modes"`](../specs/003-seed-distribution/contracts/publish-pipeline.md),
   a primary failure aborts the scenario; a secondary
   failure is treated as job-level failure (the loop
   re-raises via `set -e`).

### Hermeticity

`BAKER_COMMIT_SHA7` is the only required runtime input
that derivations cannot supply themselves. The app
exits non-zero with a clear error when it is missing or
not 7-hex. The CI `seed-image-publish` job (T006) will
export it from `git rev-parse --short=7 HEAD`.

`--dry-run` prints the four lines the contract requires
(two scenarios × two tags) and exits without invoking
`skopeo`. No env-vars beyond `BAKER_COMMIT_SHA7` are
read, so the dry-run is reproducible from any worktree.

### `flake.nix` and `justfile`

- `flake.nix` imports `nix/seed-publish.nix` with the
  same `scenariosDir`/`scenarioFiles`/`seedImage`
  threading as the rest of the seed-distribution flake
  expressions, and exposes
  `apps.<system>.publishSeedImages`.
- `justfile` adds a `publish-seed-images` recipe that
  wraps `nix run .#publishSeedImages` with a fallback for
  `BAKER_COMMIT_SHA7` (`git rev-parse --short=7 HEAD` if
  the env var is unset). Forwards positional args (notably
  `--dry-run`).

### Validation

- `nix run .#publishSeedImages -- --dry-run` (with
  `BAKER_COMMIT_SHA7` exported) prints exactly four
  lines per the contract.
- Calling without `--dry-run` against a non-existent
  registry returns non-zero visibly: `set -e` plus
  `skopeo`'s non-zero exit on auth/network failures
  propagate out of the loop.
