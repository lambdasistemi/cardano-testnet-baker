## ready

Round 2 of the determinism-bug fix — applies the narrowing
the maintainer chose (option 1 of the three I surfaced) on
top of the diagnostic round (`b4672c3`).

### What the diagnostic surfaced

CI run
[25581476450](https://github.com/lambdasistemi/cardano-testnet-baker/actions/runs/25581476450/job/75101170207)
printed exactly two file-level differences in the
`normal` seed between independent builds A and B:

```text
seed/chain-db/volatile/blocks-6.dat
  A=466e6ed1…  B=MISSING
seed/synthesis-report.json
  A=da6195f4…  B=0128404d…
```

Both cascade from the same upstream cause: the
`db-synthesizer` produces a different *number* of
`chain-db/volatile/` blocks between runs of the 300,000-slot
`normal` scenario. `synthesis-report.json` then diverges
because `chainDb.bytes`, `chainDb.fileCount`, and
`chainDb.packagedBytes` reflect the volatile-block drift; T001's
projection only strips host-clock observation, not size accounting.

`local-fast` is fully deterministic (720 slots — below whatever
threshold triggers the synthesizer drift) and the existing
`example-bake-determinism` Feature 002 check has always passed
for it.

### Maintainer's decision

I surfaced four options — fix in this PR, narrow gate, strip
chain-db, or drop `normal` from publishing — and the maintainer
picked **narrow gate, file ticket, document the limitation**. The
synthesizer drift is outside Feature 003's surface; widening this
PR's scope to fix it would push merge time out indefinitely with
uncertain return.

### What changed

1. **Issue filed:**
   [#15: `fix(synthesis): normal scenario db-synthesizer output non-deterministic across runs`](https://github.com/lambdasistemi/cardano-testnet-baker/issues/15).
   Body lays out the observed diff, the hypothesised cause, the
   reason `local-fast` doesn't catch it, the narrowed-gate
   workaround in this PR, and the resolution paths. Added to
   the Planner board with `Category=Antithesis`,
   `Ownership=Work`, no `Status` (backlog), per the same
   convention as #14.

2. **`nix/checks.nix`:** the `seed-image-determinism` check
   now iterates a `determinismScenarioFiles =
   builtins.filter (f: f == "local-fast.json") scenarioFiles`
   list rather than `scenarioFiles` directly. The comment
   block above the filter explains the limitation, links #15,
   and tells the next maintainer exactly how to widen the gate
   once #15 is resolved (revert the filter, drop the comment).
   The `mkSeedImagePair` and `determinismCheckScript` logic —
   including the per-file diagnostic added in round 1 — is
   unchanged.

3. **Spec FR-006:** appends a "Known limitation" paragraph
   linking #15 and naming the scope (gate runs against
   `local-fast` only; `normal` is still published).

4. **Spec SC-002:** appends a one-sentence pointer to the
   FR-006 limitation so the success-criterion description is
   not optimistic about what CI currently asserts.

5. **`contracts/publish-pipeline.md §"Determinism check"`:**
   adds a "Known limitation" subsection under the rationale,
   matching the spec's wording and naming the
   one-line widening in `nix/checks.nix`.

### Why this is not a T002 amend

Same rationale as the round-1 diagnostic commit: T002 is now
buried under T003, T004, T005, T006, T007, T008, plus the
diagnostic round (`b4672c3`). Amending T002 in place would
force fresh review of all of those (eight already-approved
patches). This narrowing change is small and self-contained, so
it lands as a new vertical commit on top of the diagnostic.
