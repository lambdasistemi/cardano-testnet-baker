### What changed

`specs/003-seed-distribution/contracts/seed-image-layout.md` —
the **Timestamp contract** section is rewritten to be honest
about what the artifact actually carries:

- The image config carries `created = "1970-01-01T00:00:00Z"`.
  This is set explicitly by `mkSeedImage` and verifiable via
  `skopeo inspect docker-archive:<archive> | jq .Created`.
- The customisation layer's tar entries carry mtime
  `1980-01-01T00:00:00Z`. `dockerTools.streamLayeredImage` packs
  layers with `tar --mtime="@$SOURCE_DATE_EPOCH"`, and the
  `SOURCE_DATE_EPOCH = 315532800` floor that nixpkgs uses for
  pre-DOS-epoch safety lands tar mtimes at the start of 1980.

Both timestamps are *fixed per build* — host wall-clock time
never leaks into either, so two rebuilds of the same scenario
produce byte-identical layer tars and byte-identical manifest
digests. The contract now states this explicitly: any value
other than the two above is a determinism bug.

### Why option B (contract update) and not option A (force epoch zero)

The reviewer offered both. The `mtime` parameter on
`buildLayeredImage` / `streamLayeredImage` controls only the
image config's `created` (already pinned to epoch zero in the
implementation), not the layer tar entries. Layer mtimes come
from `SOURCE_DATE_EPOCH` in the build environment, which
dockerTools' `symlinkJoin` and `extraCommands` flow sets to
`315532800` upstream. Forcing epoch-zero layer mtimes would
require either patching `dockerTools` (out of scope) or
re-tarring the layer in a post-build hook, which would diverge
from the canonical dockerTools artifact and break the
determinism check we add in T002. Documenting the actual
deterministic value preserves determinism (the property that
matters) without contorting the implementation.
