## ready

Round 2 of T004 — addresses the round-1 verdict on `4ce8728`.

The reviewer had two valid concerns:

1. The `seed-image-acceptance` flake output existed but CI
   didn't actually invoke it; the GHA job duplicated the
   build-and-run steps inline, so the flake check could
   silently drift away from what CI ran.
2. The wrapper accepted exactly one scenario name; T004's
   contract is "for every committed scenario", which made
   the wrapper insufficient to act as the single source of
   truth.

### What changed

- **`nix/checks.nix` (`seed-image-acceptance`):** the
  wrapper now iterates the same `scenarioFiles` set the
  rest of the flake enumerates, by default. Adding a new
  scenario JSON to `examples/scenarios/` immediately
  extends the acceptance coverage with no CI yaml change.
  Optional positional args still let a caller target a
  subset (`./result/bin/seed-image-acceptance local-fast`).
  The bash list of scenarios is materialised at flake
  evaluation time via `pkgs.lib.concatMapStringsSep` so
  the wrapper does not need to know about flake outputs at
  runtime.

- **`.github/workflows/ci.yml`:** the
  `compose-acceptance` job no longer duplicates the
  build/readlink/run sequence. Two steps now:
  1. `nix --quiet build .#checks.x86_64-linux.seed-image-acceptance --out-link result-seed-image-acceptance` — this builds the
     wrapper (no Docker needed); the runner installs every
     dep the wrapper declares (skopeo, jq, gnutar,
     coreutils) automatically through Nix.
  2. `./result-seed-image-acceptance/bin/seed-image-acceptance` — runs the
     wrapper, which iterates every committed scenario and
     drives `compose/acceptance/run.sh` against each
     scenario's seed-image archive.
  The standalone "Install acceptance dependencies" `apt-get`
  step is gone — Nix handles all dependencies via the
  wrapper's `runtimeInputs`.

### Why the wrapper still lives under `checks` rather than `apps`

T004's task wording is "expose it as a flake check"; `apps`
would have allowed `nix run .#seed-image-acceptance` (one
command instead of build + execute), which is slightly more
ergonomic for local use. I kept it under `checks` to match
the task. If the reviewer prefers `apps` for ergonomic
reasons, mirror it there in a follow-up — the wrapper
itself is unchanged either way.
