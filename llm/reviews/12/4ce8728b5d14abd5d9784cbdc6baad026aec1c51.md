### What

`.github/workflows/ci.yml` (`compose-acceptance` job):

- "Install acceptance dependencies" now installs `skopeo`
  alongside `jq` (T003's archive-mode path needs both).
- "Bake examples" is replaced by "Build seed images":
  `nix build .#seedImage-local-fast` and
  `.#seedImage-normal`, with stable out-links so
  `readlink -f` resolves the materialised tarball.
- The two acceptance steps now invoke
  `compose/acceptance/run.sh <scenario>
  docker-archive:<archive>` â€” the archive-mode path T003
  added â€” instead of the dir-mode form. Behaviour for the
  startup probe and `verdict=accepted` semantics is
  identical, only the input is different.

`nix/checks.nix`:

- New `seed-image-acceptance` flake check, implemented as
  `pkgs.writeShellApplication`. Given a scenario name it
  runs
  `nix build .#seedImage-<scenario> --out-link result-...`
  followed by
  `compose/acceptance/run.sh <scenario>
  docker-archive:<archive>`. Runtime inputs are
  `coreutils`, `gnutar`, `jq`, `skopeo`.
- Intentionally NOT added to the Build Gate's
  `nix build` invocation: the wrapper itself builds with
  no Docker access, but actually running it requires Docker,
  so the host must be the `compose-acceptance` runner
  (`runs-on: ubuntu-latest`), not the Build Gate runner
  (`runs-on: nixos`). Maintainers can reproduce the CI flow
  locally with
  `nix build .#checks.x86_64-linux.seed-image-acceptance &&
  ./result/bin/seed-image-acceptance local-fast`.

### Notes for the reviewer

- The wrapper-as-flake-check pattern (rather than a
  `runCommand`-based check) is the only way to expose the
  acceptance flow as a flake output without forcing the
  Build Gate to attempt to run it. Building the wrapper
  produces `result/bin/seed-image-acceptance`; the binary
  is what runs the actual flow when invoked outside the
  build sandbox.
- `compose-acceptance` job's `runs-on` and steps were
  the only structural changes in CI; `Build Gate`,
  `scenario-schema`, `bake-determinism`, `unit-tests`,
  `formatting`, `hlint` are untouched.

## changes-requested

- `nix/checks.nix:247` / `.github/workflows/ci.yml:99`: the new
  `seed-image-acceptance` flake output is not actually the CI driver.
  T004 says to expose the Docker-required flow as a flake check and
  invoke it directly from the existing `compose-acceptance` job while

  building and invoking `.#checks.x86_64-linux.seed-image-acceptance`.
  That leaves the new flake output unexercised by CI and lets the local
  reproduction command drift from the job it is supposed to mirror.

  <scenario> docker-archive:<built-archive>` for every committed
  scenario. Please make the flake-exposed acceptance driver cover the
  repository's `scenarioFiles` set by default, or otherwise have the CI
  invocation exercise the wrapper for every committed scenario so the
  check's contract and the job's behavior stay aligned.
