## ready

Round 1 of a follow-up fix to T006: the key-mode round
(`10e8b74`) flipped Compose acceptance green for both
scenarios, but exposed yet another downstream gate
failure — `Publish seed images` aborted before any
network I/O with:

```text
time="..." level=fatal msg="Error loading trust policy:
  no policy.json file found at any of the following:
    \"/run/github-runner/runner-3/.config/containers/policy.json\",
    \"/etc/containers/policy.json\""
jq: error: Could not open file /tmp/.../skopeo-dir/manifest.json
expected exactly one layer in archive, got
```

CI run:
[25596932952](https://github.com/lambdasistemi/cardano-testnet-baker/actions/runs/25596932952/job/75144622419).

### Root cause

`skopeo copy` requires a *trust policy* file before it
will run any operation. The default search path is
`/etc/containers/policy.json`, then
`$HOME/.config/containers/policy.json`. The nixos
runner has neither, so `skopeo` refuses to operate. The
publish app's `extract_scenario_digest` then doesn't
receive a manifest.json from the `dir:` form, and the
"layer count" jq read fails with the empty-string error.

T006 (`21ee706`) configured the auth file but not the
trust policy. Locally I'd never noticed because skopeo
silently picks up either canonical path on developer
hosts — but a fresh nixos runner is a clean environment.

### What changed

`.github/workflows/ci.yml`, in the `seed-image-publish`
job's auth step (renamed to "Configure skopeo (auth +
trust policy)"):

- Existing auth file write is unchanged.
- After the auth file lands, write a permissive trust
  policy at `$HOME/.config/containers/policy.json`:
  `{"default": [{"type": "insecureAcceptAnything"}]}`.
- Comment block notes that this is permissive on
  purpose for v1; image signing (and the matching
  Sigstore-aware policy) is the cosign follow-up
  tracked under
  [issue #14](https://github.com/lambdasistemi/cardano-testnet-baker/issues/14).

The publish app and `nix/seed-publish.nix` are
unchanged.

### Why this is a follow-up commit, not a T006 amend

Same rationale as the prior follow-up rounds: T006 is
buried under T007, T008, the diagnostic round, the
narrow-gate round, and the key-mode round. The trust
policy fix is small and self-contained, so it lands as
a new vertical commit on top of the stack instead of
forcing fresh review of every downstream patch.

### Validation

- The previous round (key-mode fix) showed the

  This change unblocks that pre-flight.
- The publish app's behaviour past the trust-policy

- Local validation caveat unchanged: the CI-vs-local

### Notes for the reviewer

- I considered making the publish app self-sufficient by
  having `writeShellApplication` create a temp policy
  file under `$RUNTIME_DIRECTORY` and pass
  `--policy <path>` to every `skopeo` invocation. That
  works too, and slightly tightens the per-invocation
  blast radius, but it would mean three more
  `--policy` args to maintain in
  `nix/seed-publish.nix` and a corresponding tmpfile
  cleanup. The job-level approach (write once, run
  once) is the smaller patch and matches how the auth
  file is already managed.
- The permissive policy is provisional. When #14 lands
  (cosign keyless signing), the policy will become
  `{"default": [{"type": "sigstoreSigned", ...}]}`-
  shaped, and `skopeo copy --sign` will use the
  workflow's OIDC token. That work explicitly does not
  belong in this PR.
