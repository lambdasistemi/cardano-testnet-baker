## ready

Round 1 of a follow-up fix for compose-acceptance: the
narrow-gate round (`a9dd69a`) flipped Build Gate green
but exposed a different bug downstream — Compose
acceptance failed when running against the seed-image
archive form because cardano-node refused the
`vrf.skey` file's permissions:

```text
cardano-node-1  | OtherPermissionsExist "/assets/pools/pool-a/keys/vrf.skey"
```

CI run:
[25596803710](https://github.com/lambdasistemi/cardano-testnet-baker/actions/runs/25596803710/job/75144066974).

### Root cause

T001's `extraCommands` in `nix/seed-image.nix`
normalises every file in `/seed/` to `0644`:

```bash
find seed -type f -exec chmod 644 {} +
```

The baker, by contrast, emits private-key files at
`0600` natively (verifiable on the dir-mode bake output
under `tmp/ci-acceptance/bakes/local-fast/pools/pool-a/keys/*.skey`).
Pre-T004 (`compose-acceptance` job ran the dir-mode
form against the bake), cardano-node was happy because
the keys were already `0600`. T004 moved the job to the
archive-mode form, which feeds the `0644` keys from
the image into the runtime tree, and cardano-node
correctly refused.

The seed-image-layout contract
(`specs/003-seed-distribution/contracts/seed-image-layout.md`
§"File-mode contract") already documents the expectation:

> Consumers that need the stricter `0600` mode for
> private keys (KES, VRF, cold, stake) must `chmod
> 0600` after `COPY --from=`.

The acceptance harness *is* the consumer for the
in-PR Docker Compose flow, so it should honour the
contract instead of relying on the consumer to do it
post-copy.

### What changed

`compose/acceptance/run.sh`, after copying
`baked_output_dir` into `runtime_dir` and patching the
system-start, runs:

```bash
find "$runtime_dir" -type f -name '*.skey' -exec chmod 0600 {} +
```

A short comment block points at the contract clause and
notes that dir-mode inputs (which already have `0600`)
make the chmod idempotent.

### Why this is a follow-up commit, not a T003 amend

Same rationale as the determinism diagnostic +
narrowing rounds: T003 is now buried under T004, T005,
T006, T007, T008, the diagnostic round, and the narrow
round (seven downstream approvals). The chmod is a
small, self-contained fix that lands cleanly as its
own vertical commit on top of the stack.
