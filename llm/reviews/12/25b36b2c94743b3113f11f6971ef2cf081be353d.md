```diff
     }

```

The `layer.tar` itself is byte-identical between builds
(`1ac081…/layer.tar` in both manifests). What differs is the
*OCI config*'s `history[].comment`: `dockerTools.streamLayeredImage`
embeds the customisation-layer's Nix-store path there as
informational metadata. With the round-2 `derivationSuffix`
threading making the two builds genuinely independent, those
store paths *must* differ by construction — yet the layer
payload, and every consumer-visible config field, are
byte-identical.

This is fundamentally why T002's spec wording ("compare
`.config.digest`") is too strong under the reviewer's round-1
correctness requirement. The right semantics: verify the
*layer payload* is byte-identical (this is what protects
against host-clock or hostname leaks in the customisation
layer) and verify the image config's *meaningful* fields are
byte-identical (everything except `history`, which is
informational and unavoidable for two separate Nix store
paths).

### Round 5 check structure

1. **Layer-payload equality**: assert each image carries
   exactly one layer, then `sha256sum` the two extracted
   `layer.tar` files. Mismatch ⇒ host-clock / host-name leak
   in the customisation layer (the case the reviewer's
   round-1 critique was specifically about). A single-layer
   image is what `mkSeedImage` produces, so the check also
   guards against an accidental future split into multiple
   layers that would skip part of the seed payload from the
   comparison.
2. **Meaningful-config equality**: extract the OCI config
   from each archive, strip the `history` field with
   `jq 'del(.history)'`, and `diff -u` the canonicalised
   results. Anything outside `history` is consumer-visible
   and must be deterministic.
3. **Fixed-value invariants**:
   `architecture == "amd64"`, `os == "linux"`, and the
   `created` field starts with `1970-01-01T00:00:00`
   (allowing the `+00:00` UTC suffix dockerTools emits).
   Mutating any of these in `nix/seed-image.nix` flips the
   assertion, satisfying the T002 acceptance criterion.

### What changed

The reviewer's critique on `nix/checks.nix:18` was correct:
`base.overrideAttrs (old: { name = ...; })` only flipped the
*outer archive wrapper*'s `runCommand` name. The inner
`streamLayeredImage` derivation (the one that actually produces
the layer tar) was already captured inside `base`, so `imageA`
and `imageB` shared the same layer-producing stream — the diff
was tautological and the check could not catch host-clock leaks
in the customisation layer.

Round 2 makes the two builds genuinely independent by threading
a `derivationSuffix` parameter through `mkSeedImage`:

- `nix/seed-image.nix`: adds `derivationSuffix ? ""` to
  `mkSeedImage`. The suffix is appended to the *bakeOut*
  `runCommand`'s name, so two calls with different suffixes
  produce two `bakeOut` store paths (with byte-identical
  contents). Because `extraCommands` captures `${bakeOut}`
  literally, the two `streamLayeredImage` derivations now have
  distinct hashes and rebuild from scratch — including the
  customisation layer. The OCI image's `name`, `tag`, `created`,
  `architecture`, and `config` are unchanged, so the *output*
  tar bytes (and the manifest digest) must match across the
  pair when the implementation is deterministic.

- `nix/checks.nix`: drops the `overrideAttrs` trick and uses
  `derivationSuffix = "-determinism-a"` /
  `derivationSuffix = "-determinism-b"` on two distinct
  `seedImage.mkSeedImage` calls. The skopeo-inspect comparison
  is now a real determinism oracle: the check fails if the
  customisation layer carries any byte that varies between
  builds.
