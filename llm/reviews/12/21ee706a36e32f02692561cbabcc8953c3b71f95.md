### What

`.github/workflows/ci.yml` (new `seed-image-publish`
job):

- `runs-on: nixos` — has Nix, can invoke `nix run`
  cheaply via cachix.
- `needs: [build-gate, compose-acceptance]` — never
  runs unless the determinism gate AND the
  build-image-then-acceptance flow (T004) are both green.
  Per
  [contract `publish-pipeline.md §"Failure modes"`](../specs/003-seed-distribution/contracts/publish-pipeline.md):
  acceptance failure ⇒ no skopeo invocation, ever.
- `permissions: { contents: read, packages: write }` at
  the job level — only this one job gets registry-write
  scope; the rest of the workflow keeps the default
  read-only `packages` perm.

Two steps:

1. **Authenticate skopeo against ghcr.io.** The job
   builds a Docker-style `~/.docker/config.json` with
   `{ auths: { "ghcr.io": { auth: <base64-of-actor:token> } } }`
   and exports `SKOPEO_AUTH_FILE` to that path via
   `$GITHUB_ENV`. `skopeo` honours
   `SKOPEO_AUTH_FILE` (and falls back to
   `~/.docker/config.json`); writing both gives
   resilience against any future env-var rename.
2. **Publish seed images.**
   `BAKER_COMMIT_SHA7` is set to the first 7 chars of
   `${{ github.sha }}` (a full 40-hex SHA), then
   `nix --quiet run .#publishSeedImages` pushes both
   tags for every committed scenario via the T005 app.

### Validates / acceptance

- Spec FR-008 ("publish flow MUST run on every push to
  main and on every PR branch"): the new job is in the
  workflow, depending only on already-required jobs; no
  branch filter at the job level.
- The job pushes four tags total per push (two scenarios
  × two tags). On re-runs, `skopeo copy` is idempotent
  and re-attaches the same content-addressable manifests
  to the same tags.
- Authentication path uses
  `base64` of `<actor>:<token>` — the canonical
  Docker / OCI-distribution auth header form. The
  pre-existing `hlint` step's auth model (which uses
  `cachix-action` for cache only) is unchanged.

### Why `SKOPEO_AUTH_FILE` rather than skopeo --creds

Putting credentials on the command line via
`skopeo --creds` would leak the token into job logs
the next time someone enables `set -x`. The auth-file
form keeps the secret in a 0600 file and lets skopeo
honour it idiomatically (the same way Docker reads
`config.json`).
